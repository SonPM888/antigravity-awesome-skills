# AG Operating Policy

> Rules governing how AG (Antigravity AI) creates, names, stores, and tracks files.
> All AG-generated content MUST follow these rules.
> Policy has 2 tiers: **Universal** (all workspaces) + **Context** (auto-detect by workspace).

---

# TIER 1 — UNIVERSAL RULES (All Workspaces)

## §1. File Naming Convention

```
[YYYYMMDD]_AG_[Type]_[Description]_v[N].[ext]
```

| Type Code | Description | Example |
|-----------|-------------|---------|
| `Report` | Reports | `20260222_AG_Report_Revenue_Q4_v1.docx` |
| `Draft` | Draft docs/emails | `20260222_AG_Draft_Reply_CFO_v1.txt` |
| `Analysis` | Data analysis | `20260222_AG_Analysis_Market_Trend_v1.xlsx` |
| `Extract` | Data extractions | `20260222_AG_Extract_Email_Summary_v1.md` |
| `Minutes` | Meeting minutes | `20260222_AG_Minutes_HDQT_K1_v1.docx` |
| `TTr` | Proposals (Tờ trình) | `20260222_AG_TTr_Mua_sam_CNTT_v1.docx` |
| `NQ` | Resolutions (Nghị quyết) | `20260222_AG_NQ_HDQT_Phat_hanh_v1.docx` |
| `CBTT` | Public disclosures | `20260222_AG_CBTT_NQ_Nhan_su_v1.docx` |
| `Template` | Document templates | `AG_Template_Meeting_Minutes_v1.docx` |
| `Log` | Activity logs | `AG_Activity_Log_202602.csv` |

## §3. Archive-Only Policy

> [!CAUTION]
> **AG MUST NEVER delete any file.**
> - Draft completed → move to `Archive/`
> - Chairman decides deletions manually
> - All logs are append-only, never modified
> - Status changes = new entry, not overwrite

---

## §4. Safety Rules

| Rule | Detail |
|------|--------|
| No send without approval | Always draft → Chairman review → send |
| No delete | Archive only |
| Immutable logs | Logs cannot be edited/deleted |
| No exposed secrets | API keys, tokens → .env only, never commit |

---

## §5. Error Handling

```text
If API/service fails:
1. Retry 2x with 5s delay
2. If persistent → partial result with ⚠️ flag
3. Tell Chairman which service is down
4. Continue with available data
```

> [!IMPORTANT]
> All workflow actions MUST be logged. Logs are append-only, immutable, timestamped.

| Log Type | Format | Purpose |
|----------|--------|---------|

| Kaizen log | `kaizen_log.csv` | Self-review: wins, friction, rework |

These logs exist in **every workspace** and are consumed by `/PMSkaizen` or `/BSCkaizen`.

## §P2. State Management — Multi-File Model

PMS state is distributed across linked documents with **cross-referencing** (learned from BSC blockchain model):

### Task ID Convention

Every sprint task MUST have a stable ID: `S{sprint}.{task}` (e.g. `S1.3`).
All PROGRESS entries, commit messages, and briefing cards MUST reference these IDs.

```text
✅ "Completed S1.3: meta_pipeline.py orchestrator"
❌ "worked on pipeline stuff"
```

```text
ROADMAP.md ←──────── Single entry point (§0. Quick Status)
  ↑ updated by          ↓ read by
PMSaudit              PMSdaily (next morning)
PMSquit

PROGRESS.md ←──────── Session log + briefing cards
  ↑ updated by          ↓ read by
PMSquit               PMSdaily (context reload)
PMShotfix

BUGS.md ←──────────── Bug + known issue tracker
  ↑ updated by          ↓ read by
PMSaudit              PMSdaily (open FAIL items)
PMShotfix

README.md ←──────────── Project status line
  ↑ updated by
PMSaudit
```

## §P3. PMS Daily Rhythm

```text
Open IDE
  ↓
/PMSdaily ← Context reload, Sprint briefing, pick task
  ↓
Work ← Execute tasks from ROADMAP sprint
  ↓
Milestone? → /PMSaudit (syncs all docs + git commit)
  ↓

  ↓
```

## §B2. State Management — Blockchain/Ledger Model

### BSC_STATE.json — Single Source of Truth

- **Location:** `AG_Output/Logs/BSC_STATE.json`
- **Purpose:** All BSC workflows READ and WRITE this single file
- **Model:** Append-only ledger with sequential numbering

### Immutability Rules

| Rule | Detail |
| ---- | ------ |
| Never delete | No ledger entry may be edited or removed |
| Never overwrite | Status changes = new entry with `ref_seq` pointing to original |
| Sequential numbering | Each entry gets `seq: N` (auto-increment) |
| Timestamped | ISO 8601 with timezone on every entry |
| Full traceability | Any item traceable from creation → every state change → close |

### Active View (computed by BSCbrief)

Filter ledger for items where latest status ≠ `closed`/`confirmed`/`archived`. This is the working view — never stored separately.

### Data Flow

```text
BSCboard ──WRITE──→ ┌─────────────┐ ←──READ── BSCbrief (Step 0)
BSCaudit ──WRITE──→ │ BSC_STATE   │ ←──READ── BSCkaizen (metrics)
BSCwatch ──WRITE──→ │  .json      │
BSCdisclosure─UPD─→ │ (ledger[])  │
BSCdraft ──WRITE──→ │             │
BSCkpi ────WRITE──→ └─────────────┘
```
